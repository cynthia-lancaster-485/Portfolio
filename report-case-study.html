What changed and why

Every decision had a reason.

Before — Pain points

  • 18+ column table forced constant horizontal scrolling
  • No saved reports — users rebuilt the same config every session
  • Period picker, filters, and run action scattered with no hierarchy
  • All state visible at once with no progressive disclosure
  • Dense striped tables and tiny fonts increased cognitive load
  • No clear confirmation of what was selected before running

After — Design decisions

  • My Reports page lets users save and reuse configurations in one click
  • Clicking a saved report goes straight to the runner — no extra steps
  • Two-column layout mirrors the real workflow: pick report → configure → run
  • Parameters appear inline below the report list — contextual and clear
  • Entity table trimmed to essential columns; scrollable overflow preserved
  • Breadcrumb navigation makes it easy to get back without losing context

What I learned

Three things this project taught me.

01

Know your power users

The original design was built for admins who needed every field. But most users just needed name, type, and code. Designing for the common case — not the edge case — made everything simpler.

02

Save state, save time

The biggest UX win wasn't visual — it was saved report configurations. Users were manually re-selecting the same companies and reports every single day. One feature, massive impact.

03

My dev background is a UX asset

Because I could read the TypeScript, I understood exactly what state existed and how it flowed. That let me design a UI that matched the data model instead of fighting against it.
