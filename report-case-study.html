The problem

The original system made a simple task feel exhausting.

Users needed to select reports and companies, set a period, and run them — a straightforward workflow buried under an overwhelming wall of controls and data. The interface prioritized showing everything at once over helping users accomplish anything quickly.

20+

Columns in the company table

The main selection table showed every data field at once, forcing users to scroll horizontally to find what they needed.

0

Saved report configurations

Every session started from scratch. Users had to re-select the same reports and companies over and over, every single time.

∞

Context switching required

Controls for period, search filters, company list type, and run action were all scattered, with no clear visual hierarchy.

What changed and why

Every decision had a reason.

Before — Pain points

  • 20-column table forced horizontal scrolling to find any company
  • No saved reports — users rebuilt the same config every session
  • Period, filters, and run button scattered with no visual grouping
  • All state visible at once with no progressive disclosure
  • Dense table striping and small fonts increased cognitive load
  • No confirmation of what was selected before running

After — Design decisions

  • My Reports page lets users save and reuse configurations in one click
  • Company table trimmed to 5 essential columns — full data still accessible
  • Period selector moved to a persistent top bar — always visible, never lost
  • Two-step flow (select → run) mirrors the user's actual mental model
  • Selected items shown as chips in a dedicated column — clear confirmation
  • Card layout for saved reports makes scanning and acting much faster

What I learned

Three things this project taught me.

01

Know your power users

The original design was built for an administrator who needed every field. But 80% of users just needed name, type, and code. Designing for the common case, not the edge case, made everything simpler.

02

Save state, save time

The biggest UX win wasn't visual — it was the saved report configuration. Users had been manually re-selecting the same 10 companies and 3 reports every day. One feature, massive daily time savings.

03

My dev background is a UX asset

Because I could read the TypeScript, I understood exactly what state existed and how it flowed. That let me redesign the UI to match the data model — not fight against it.
